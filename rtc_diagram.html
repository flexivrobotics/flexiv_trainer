The problem

Naïve replan — chunks disagree at the seam

The robot is mid-way through the old chunk when a new observation arrives and a new chunk is denoised from independent random noise. It knows nothing about the path in flight, so its first waypoint can land far from where the robot was heading. Replacing the pending waypoints there produces a discontinuity — the jerk.

Root cause. Two trajectories sampled from unrelated noise share no continuity constraint. A post-hoc trajectory blend can't fix it — it just averages two dynamically-inconsistent paths.
↓ RTC rewrites the cycle ↓
Overview

The overlapped cycle

Inference takes time. While the new chunk is computing, the robot keeps executing the old one — so by the time the new chunk lands, the first d steps of it are already in the past. RTC freezes exactly those d steps onto the still-executing tail, so the new chunk continues the old one instead of contradicting it.

d = inference delay. How many action steps elapse during inference (infer_ms / dt). s = execution horizon — where the blend from old→new finishes. Constraint: d ≤ s ≤ H − d.
Inside one inference

Denoising loop with a frozen prefix × 16 steps

A fresh chunk starts as pure noise and is denoised over your 16 DDIM steps. At every step, RTC re-asserts the prefix: it re-noises the clean old-tail to the current noise level, blends it in under the fade weights, then lets the black-box model denoise one step.

The model never knows. It always receives a chunk uniformly noised to level t — it can't tell the prefix was planted. That's why this works on an unmodified, already-trained policy.
Zoom · the re-noise

Lifting the clean prefix up to noise level t

The rest of the sample sits at noise level t, but the prefix is clean (level 0). To make them compatible, jump the prefix up to the same level in one closed-form move — the diffusion forward formula, no model, no loop:

noisy_prev = √(ᾱt)·prev_actions + √(1−ᾱt)·target_noise  — ᾱt is a precomputed constant; target_noise is sampled once per chunk and reused every step so the prefix rides the same latent path.

The blend

Fade weights — hard freeze, then hand off

Per position across the horizon: weight 1.0 = take the old path exactly; weight 0.0 = fully freshly planned. The first d steps are pinned hard; from d to s the weight decays, handing control smoothly to the new plan.

Two schedules. exp hands off fast (steeper); linear is gentler. Tunable via rtc_prefix_schedule.
Result

Chunks that weld instead of clash

The new chunk now leaves the old one exactly where the robot is, and only diverges once past the frozen head. Executed end-to-end, replans become invisible — one continuous path.
