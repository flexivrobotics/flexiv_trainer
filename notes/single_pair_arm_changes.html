Single-Pair Arm Support

Let the operator choose a single pair or a dual pair of arms from the Web UI, threaded end-to-end.

Repo flexiv_trainer · branch feature/single_arm_pair · This document tracks the change set file-by-file. Sections are marked Done (committed to the working tree) or Planned (from the approved plan, not yet written).

Problem

What this change is trying to solve.

flexiv_trainer is an industrial teleoperation + dataset-recording + training app (FastAPI backend, vanilla-JS frontend, Pydantic config). Today it assumes exactly two leader/follower arm pairs — a "dual pair" / bimanual rig. That "exactly 2" is hardcoded in the config defaults, the runtime manager, and the Web UI.

Operators who run a single pair of arms have no way to use the app: the serial inputs, teleop loop, recording feature set, and UI all assume two arms. The goal is to let the operator choose single or dual from a dropdown in the Web UI, and have that choice drive teleop, recording, and the interface end-to-end — with dual remaining the default so existing setups are untouched.

Implementation Plan

The original plan: the model of the setting and the architectural finding that shapes the work.

Decisions (confirmed with user)

  • Single-arm side is configurable: in single mode a second dropdown picks whether the lone arm is left_arm/left_wrist or right_arm/right_wrist (default right). It only appears when "Single pair" is selected.
  • Selector lives on the Home config page and is persisted alongside robot serials in .local/robot_serials.json.
  • Default = dual; single is opt-in. Backward compatible.

Model of the setting

Two fields on the persisted RobotSerialConfig, plus a derived single source of truth that replaces every hardcoded 2 / _RECORDING_SIDES:

Modeactive_arm_count()active_sides()
dual2["left_arm", "right_arm"]
single + right1["right_arm"]
single + left1["left_arm"]

Key architectural finding (reduces scope)

The data/recording layer is already arm-count agnostic at runtime:

  • _iter_combined_features() enumerates whatever robots are in the snapshot — it does not assume 2.
  • arm_side_label(index) already maps 0→left, 1→right with a generic fallback.
  • TeleopService already iterates _get_robot_pairs() and filters by configured serials — it works with 1 pair as-is.

So the real work is three-fold, and reuses the existing serial-persistence, debounced-save, and recording-option machinery rather than inventing new plumbing:

  1. Add an arm_mode concept to config (with the derivation helpers above).
  2. Make the parts that hardcode 2 / ["",""] / ["left_arm","right_arm"] respect that mode.
  3. Surface the dropdown(s) in the UI and thread the value through the save path.
One genuinely new bit of logic: because features are keyed by capture-order index, a single arm sits at index 0 and would otherwise label as left_arm. The chosen side from single_arm_side must override that mapping, so a single right arm records as right_arm.

Overview

What the feature does and the core design decisions.

  • Add an arm-mode selector on the Home page: Single pair or Dual pair. Default is dual, so existing setups are unchanged.
  • In single mode, a second dropdown chooses which side the lone arm is — Left or Right (default Right) — driving its recording side label and wrist camera.
  • The selection is persisted alongside robot serials in .local/robot_serials.json.
  • A single source of truth — active_arm_count() / active_sides() on RobotSerialConfig — replaces the hardcoded "exactly 2" assumptions across the stack.
Why the scope is small: the data/recording layer is already arm-count agnostic at runtime (_iter_combined_features enumerates whatever robots exist; arm_side_label handles indices generically). The hard "2" lives in config, the runtime manager, and the UI.
