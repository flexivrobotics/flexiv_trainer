Performance Diagnosis/ flexiv_trainer/ WebUI · depth preview

Depth previews stall the whole WebUI while they decode

Depth frames are the only images decoded on the server. The decode work runs in the same single, GIL-bound process that answers every poll and live camera request — so opening a depth preview starves the entire UI, not just the depth panel.

Root cause confirmed Server-side, single process FastAPI + uvicorn No code changed

The one-sentence verdict

Every backend endpoint is a synchronous def sharing one 40-thread pool inside a single GIL-bound process. Depth previews are the only images decoded server-side (RGB plays natively in the browser), so the whole-clip prewarm thread plus per-frame decodes saturate that shared pool and starve the 10 Hz status polls and the self-clocked live-camera feed — which is what you feel as lag.

01

Why RGB is smooth but depth is not

RGB and depth take completely different routes to the browser, and that asymmetry is the whole story.

RGB is served as an H.264 MP4 and played in a native <video> element. The browser decodes it on the GPU/media pipeline; the server just streams bytes off disk. Zero decode load on the Python process.

Depth is stored as 12-bit HEVC, which the browser cannot play. So the server must decode each frame with PyAV, dequantize it (via torch/numpy), colorize it with an OpenCV JET map, and JPEG-encode it — then hand back a single image. The frontend fetches these one JPEG per frame from /datasets/frame-image.

02

The server has no room to absorb the load

The backend is a single uvicorn process, one event loop, and one shared AnyIO threadpool (default cap ~40 threads). Critically, every route handler is a plain synchronous def — confirmed across all five routers, zero async def. That means every request — status polls, live camera frames, dataset previews, and depth decodes alike — competes for the same finite pool of threads, inside one GIL-bound interpreter.

There is no worker count, no CapacityLimiter, no ProcessPoolExecutor, and no offloading. Nothing isolates the heavy decode work from the requests that keep the UI feeling alive.

Why it feels UI-wide, not local

The frontend is a polling app — no websockets. Teleop status polls at 10 Hz, rollout at ~3 Hz, recording save at 4 Hz, plus fault and training pollers. Each hits a sync endpoint on the same pool. When decode saturates it, the heartbeat polls queue behind decodes too — so the sluggishness shows up everywhere at once.

03

The two things that compound

Thread-pool starvation

Opening a depth preview fires a background depth-prewarm thread that decodes the entire clip, while scrubbing/playback also drives per-frame on-demand decodes through the sync /datasets/frame-image endpoint. The live camera feed is a self-clocked pump — it requests the next frame the instant the previous one loads, so its FPS is entirely gated by server response time. Starve the pool and the live feed's frame rate collapses directly.

GIL contention

Even setting thread slots aside, the decode is CPU-bound Python. cv2.applyColorMap and cv2.imencode release the GIL, but dequantize_depth (torch/numpy) and the Python-level frame iteration do not — so the prewarm thread genuinely holds the interpreter, not just a pool slot. On a 16-core box the cores are there; the single interpreter is the bottleneck.

What is already mitigated (and isn't the bug)

The frontend gates img.src on load and holds only the latest requested frame, so a single client doesn't spam its own requests. There are caches too: a per-frame LRU (64 entries) and a whole-clip cache (8 clips), plus immutable browser caching on responses. And the merge job itself is cheap — a file-level copy with no re-encode. The cost is purely in previewing merged/episode data. None of these address cross-request pool and GIL contention.

04

Fixes, by impact-per-effort

1
Bound depth-decode concurrency

Wrap the decode paths in an AnyIO CapacityLimiter (e.g. 2–4 slots) so decodes can never consume all 40 threads. Guarantees headroom for the 10 Hz polls and the camera pump even mid-prewarm.

effort: small risk: low do this now
2
Move decode out of the web process

Run the decode in a ProcessPoolExecutor so the decode GIL lives in a separate interpreter. The web process only receives finished JPEG bytes — decode CPU is fully decoupled from UI responsiveness.

effort: medium risk: medium most robust runtime fix
3
Pre-decode depth to browser-native video at merge time

The merge job already touches these files — transcode/colorize depth into an H.264 MP4 then, so the browser plays depth natively via <video>, exactly like RGB. This removes the per-frame round-trip entirely and makes depth as smooth as RGB. Trades disk + one-time merge cost for zero live decode.

effort: larger risk: medium best long-term UX

Recommended path: ship #1 now for an immediate, safe fix, then take on #3 as the real cure.
